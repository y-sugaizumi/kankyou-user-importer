# アーキテクチャ

## レイヤー構成

```
┌─────────────────────────────────────┐
│   プレゼンテーション層 (ui/)        │
│   - flet_app.py (GUI)              │
│   - cli.py (CUI)                   │
└─────────────────────────────────────┘
              ↓
┌─────────────────────────────────────┐
│   サービス層 (services/)            │
│   - customer_import.py             │
│     ビジネスロジックを集約          │
└─────────────────────────────────────┘
              ↓
┌─────────────────────────────────────┐
│   データアクセス層                   │
│   - models/ (ORM)                  │
│   - schemas/ (バリデーション)       │
└─────────────────────────────────────┘
              ↓
┌─────────────────────────────────────┐
│   インフラ層                         │
│   - config/ (設定)                 │
│   - utils/ (ユーティリティ)         │
└─────────────────────────────────────┘
```

## 主要コンポーネント

### CustomerImportService

顧客インポートのビジネスロジックを担当。

**メソッド:**
- `load_from_excel(file_path)` - Excelファイル読み込み
- `check_diff(users)` - 既存データとの差分チェック
- `import_customers(users)` - データベースへの登録

### データフロー

1. **ファイル選択** (UI層)
2. **Excel読み込み** (Service層 → Utils層)
3. **バリデーション** (Schemas層)
4. **差分チェック** (Service層 → Models層)
5. **パスワードハッシュ化** (Utils層)
6. **DB登録** (Service層 → Models層)

## 設計原則

- **関心の分離**: UI、ビジネスロジック、データアクセスを分離
- **依存性の逆転**: 上位層は下位層に依存、下位層は上位層に依存しない
- **単一責任**: 各モジュールは1つの責任のみを持つ
- **テスタビリティ**: サービス層は独立してテスト可能

## 拡張性

新しいUI（例: REST API）を追加する場合:
1. `ui/api.py` を作成
2. `CustomerImportService` を呼び出す
3. ビジネスロジックの変更は不要
